# Fall — Stage 0: วัด baseline ก่อนเทรนอะไรทั้งสิ้น

> ถ้าข้ามขั้นนี้ คำว่า "โมเดลใหม่ดีขึ้น" จะเป็นความรู้สึก ไม่ใช่ข้อเท็จจริง

เครื่องมือ: [`scripts/fall_eval.py`](../scripts/fall_eval.py)

มัน replay คลิปผ่าน **`PPEEngine` ตัวจริงที่แอปใช้** (person detector เดิม, tracker เดิม,
หน้าต่าง 30 เฟรมเดิม, confirm + cooldown + spatial dedupe เดิม) แล้วรายงานว่า
**"ระบบจะเตือนไหม"** ไม่ใช่ "โมเดลให้ค่าความน่าจะเป็นเท่าไร" — เพราะสิ่งที่โรงงานเห็นคือการเตือน

---

## ⚠️ สิ่งที่ต้องรู้ก่อน: หน้าต่างเวลาไม่ตรงกับที่โมเดลถูกเทรนมา

`.env` ตั้ง `FALL_LOOP_FPS=10` → หน้าต่าง 30 เฟรมของเรากินเวลา **3.0 วินาที**
แต่โมเดล TFLite ที่ใช้อยู่ถูกเทรนมาบน 30 เฟรมที่ **~30 fps = 1.0 วินาที**

**เราป้อนการเคลื่อนไหวที่ช้ากว่าที่มันเคยเห็น 3 เท่า** — คาดหวังได้เลยว่า baseline จะออกมาไม่สวย
และนั่นคือ *ประเด็น*: มันบอกเราว่าปัญหาอยู่ที่ **cadence ไม่ตรงกัน** ไม่ใช่ที่ threshold
harness จะพิมพ์ค่านี้ให้เห็นทุกครั้งที่รัน

---

## 1) เตรียมคลิป

วางไฟล์วิดีโอไว้ที่ `backend/data/fall_clips/` (โฟลเดอร์นี้ถูก gitignore — คลิปคนจะไม่หลุดขึ้น git)

### คลิป positive (ล้ม) — โหลดจากอินเทอร์เน็ตได้

| Dataset | หมายเหตุ |
|---|---|
| **MCFD** (Multiple Cameras Fall) | ⭐ มีมุมกล้องติดเพดาน ใกล้ CCTV จริงที่สุด — **แนะนำให้ล็อกไว้เป็น test อย่างเดียว ห้ามเอาไปเทรน** |
| **Le2i** | มี annotation บอกเฟรมเริ่ม/จบการล้ม |
| **URFD** | 30 ล้ม + 40 ADL, label ต่อเฟรม |

### คลิป negative — **ถ่ายเอง ไม่มีใน dataset ไหน และสำคัญที่สุด**

Gate 8 เขียนว่า **"นั่งกับพื้น 5 นาที = 0 false positive"** — ไม่มีชุดข้อมูลสาธารณะชุดไหนมีคลิปนี้
ถ่ายเองใช้เวลารวมไม่ถึง 30 นาที และ **ไม่ต้องล้มจริง ไม่มีใครเจ็บตัว**

- นั่งกับพื้น 5 นาที ← ท่านี้คือด่านที่ยากที่สุด
- นอนลงตั้งใจ · หมอบ · คลาน · ก้มเก็บของ · ผูกเชือกรองเท้า
- เดินไปมาปกติ 10 นาที

ถ่ายจาก **มุมเดียวกับที่จะติดตั้งกล้องจริง** — มุมกล้องสำคัญกว่าจำนวนคลิป

## 2) ติดป้ายเวลา

```bash
python scripts/fall_eval.py --clips backend/data/fall_clips --init-labels
```

จะได้ `labels.json` ให้กรอกวินาทีที่เกิดการล้ม:

```json
{
  "clips": {
    "le2i_coffee_01.mp4":  { "falls": [[4.2, 5.8]] },
    "sitting_on_floor.mp4": { "falls": [] }
  }
}
```

`"falls": []` = คลิป negative · คลิปที่ไม่อยู่ใน labels.json harness จะ **ปฏิเสธไม่ยอมรัน**
(การเดาว่าคลิปที่ไม่ได้ติดป้ายเป็น negative คือวิธีที่ทำให้คลิปล้มถูกนับเป็น false positive)

## 3) รัน

```bash
python scripts/fall_eval.py --clips backend/data/fall_clips --dump-frames
```

ได้:
- **recall / precision / false-alert ต่อนาที / latency** เทียบกับ Gate 8 ทั้ง 4 ข้อ
- **คำอธิบายว่าทำไมถึงพลาดแต่ละครั้ง** — ซึ่งเป็นส่วนที่บอกว่าต้องแก้อะไร เช่น
  - *"person detector ไม่เจอคนเลย"* → ปัญหาอยู่ที่ Stage 1 ไม่ใช่ fall
  - *"โมเดลให้ p สูงสุด 0.85 < threshold 0.90"* → โมเดลมองไม่เห็น ต้องเทรนใหม่
  - *"โมเดลยิงแล้ว แต่ถูก corroboration/confirm/cooldown บล็อก"* → ปัญหาอยู่ที่ logic ไม่ใช่โมเดล
- `eval_out/alerts/` ภาพทุกครั้งที่ระบบเตือน (ดู false positive ด้วยตา — สำคัญกว่าตัวเลข)
- `eval_out/trace_*.json` ค่า p_fall / pose coverage / prone ทุก tick

## 4) ก่อนวัด — แก้บั๊ก rule layer ก่อน

ตอนนี้ [`fall_detector.py`](../backend/utils/fall_detector.py) บรรทัด `rule_fall = (coverage < min_coverage) and transition`
ทำให้ rule layer ทำงาน **เฉพาะตอนที่ pose อ่านคนไม่ได้** คนที่ล้มอยู่ชัด ๆ แต่โมเดลให้ 0.85
(ต่ำกว่า threshold) จะ **ไม่ถูกเตือนเลย** ทั้งที่กฎเห็นชัดว่าล้มแล้วนอนนิ่ง 2 วินาที

ถ้าไม่แก้ก่อน baseline จะต่ำเกินจริง แล้วเราจะเข้าใจผิดว่าโมเดลใหม่ดีขึ้น ทั้งที่จริงแค่แก้บั๊ก

**รัน harness ทั้งก่อนและหลังแก้** เพื่อแยกให้ออกว่า "ดีขึ้นเพราะแก้บั๊ก" กับ "ดีขึ้นเพราะเทรนใหม่"
เป็นคนละเรื่องกัน
