# ZENTRA — แผนเทรนโมเดลใหม่ตั้งแต่ต้น (Staged Training Pipeline)

> ทุกตัวเลขในเอกสารนี้ **วัดจากข้อมูลจริงในเครื่อง** เมื่อ 2026-07-10 ไม่ใช่การประมาณ
> คำสั่งที่ใช้วัดอยู่ท้ายเอกสาร ทำซ้ำได้

---

## ส่วนที่ 0 — สิ่งที่พบก่อนวางแผน (อ่านก่อน ห้ามข้าม)

ผมสำรวจ `backend/data/train_dataset/` แล้วพบปัญหา 3 ข้อที่ทำให้ **ผลการเทรนที่ผ่านมาทั้งหมดเชื่อถือไม่ได้**

### 🔴 ปัญหา 1 — ข้อมูล PPE ส่วนใหญ่ "ไม่เคยเห็นของอยู่บนตัวคน"

นับจาก 61,968 ภาพใน `roboflow_dl/train`:

| คลาส | จำนวนภาพ | ภาพที่มีกล่อง `person` ด้วย |
|---|---:|---|
| person | 16,220 | 100% |
| no_helmet | 3,558 | **97.0%** ✅ |
| helmet | 28,712 | **46.0%** ✅ |
| no_glasses | 668 | 8.4% |
| no_gloves | 2,176 | **1.3%** ❌ |
| no_vest | 7,684 | **0.5%** ❌ |
| Vest | 15,888 | **0.0%** ❌ |
| gloves | 2,260 | **0.0%** ❌ |
| glasses | 1,264 | **0.0%** ❌ |
| boots | 704 | **0.0%** ❌ |
| no_boots | 2,464 | **0.0%** ❌ |

ขนาดวัตถุมัธยฐาน (px ที่ imgsz 640): `glasses 260×200`, `gloves 274×332`
→ **นี่คือภาพโคลสอัพสินค้า/มือ ไม่ใช่ภาพคนในโรงงาน**

**นี่คือคำตอบว่าทำไมแว่นกับถุงมือไม่เคยตรวจเจอในกล้องจริง** ไม่ใช่บั๊กโค้ด ไม่ใช่ค่า conf
โมเดล**ไม่เคยเห็นแว่นบนหน้าคนที่ยืนอยู่ในโรงงานเลยแม้แต่ภาพเดียว** มันเห็นแต่แว่นวางบนโต๊ะ
ในกล้องจริงแว่นมีขนาดราว 10–15 px ต่างจากที่เทรนมา ~20 เท่า

ผลข้างเคียง: logic จับคู่ `area(item ∩ person) / area(item) ≥ 0.30` **ใช้กับคลาสเหล่านี้ไม่ได้เลย**
เพราะโมเดลไม่เคยเรียนรู้ว่ามันควรอยู่ในกรอบคน

### 🔴 ปัญหา 2 — Validation set รั่ว (augmentation leakage) — **และ test set ก็รั่วด้วย**

Roboflow สร้างภาพ augment 2–10 ใบต่อภาพต้นฉบับ 1 ใบ แล้วแบ่ง train/valid **ตามภาพ augment ไม่ใช่ตามภาพต้นฉบับ**

```
train↔valid : 962 ภาพต้นฉบับซ้ำกัน  (24.5% ของ valid)
train↔test  : 852 ภาพต้นฉบับซ้ำกัน  (21.6% ของ test)   ← ฉบับแรกของเอกสารนี้มองข้ามไป
valid↔test  : 135 ภาพต้นฉบับซ้ำกัน  ( 3.4% ของ test)
```

→ **mAP 0.698 ที่เคยรายงานไว้ สูงเกินจริง** เพราะโมเดลเคยเห็นภาพต้นฉบับของ 1 ใน 4 ของชุดทดสอบมาแล้ว
→ และ **ไม่มี split ไหนสะอาดเลย** ใช้ test set เป็น "ตัวตัดสินสุดท้าย" แทน valid ก็ไม่ได้เช่นกัน

วัดซ้ำได้ด้วย `python3 scripts/audit_dataset.py --root backend/data/train_dataset/roboflow_dl`

### 🔴 ปัญหา 3 — คลาสหนึ่งวัดผลไม่ได้เลย + ข้อมูลถูกทิ้งไปครึ่งหนึ่ง

- `no_boots` มี **0 instances ใน valid set** → ประเมินความแม่นยำไม่ได้ ตัวเลขใดๆ เป็นค่าว่าง
- ความไม่สมดุล **48 : 1** (helmet 43,970 vs no_glasses 918 instances)
- `merged_train.txt` ใช้เพียง **10,632 จาก 23,493 ภาพต้นฉบับ (45%)** — ที่เหลือถูกทิ้งเงียบๆ

### ⚠️ ปัญหา 4 — ไม่มีชุดทดสอบจากกล้องจริงเลย

ทุกตัวเลขที่เคยวัด มาจาก Roboflow val set ซึ่งเป็นภาพจากอินเทอร์เน็ต
**ไม่มีอะไรบอกได้ว่าโมเดลทำงานดีแค่ไหนในโรงงานของคุณ**

## สรุปข้อเสนอเชิงสถาปัตยกรรม

ข้อมูลข้างบนบอกว่า **การใส่ทุกคลาสไว้ใน detector ตัวเดียวคือการออกแบบที่ผิด**
ของชิ้นเล็ก (แว่น ถุงมือ รองเท้า) มีขนาด 10–30 px ในภาพ CCTV — detector ที่ imgsz 640 มองไม่เห็นตั้งแต่แรก

**เสนอ: two-stage**

```
                      ┌──────────────────────────────┐
  frame ─────────────▶│ Stage 1: Person Detector     │──▶ person boxes + track id
                      │ (YOLO, COCO-pretrained)      │
                      └──────────────────────────────┘
                                    │
             ┌──────────────────────┼──────────────────────┐
             ▼                      ▼                      ▼
   ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────────┐
   │ Stage 2: Zone    │  │ Stage 3-7: PPE   │  │ Stage 8: Fall        │
   │ เรขาคณิต ไม่เทรน │  │ crop คน → classify│  │ pose sequence → TFm │
   └──────────────────┘  └──────────────────┘  └──────────────────────┘
```

**ทำไม crop-classifier ถึงชนะ detector สำหรับ PPE:**

| | detector เดิม | crop classifier |
|---|---|---|
| แว่นในภาพ 640px | ~12 px → มองไม่เห็น | crop หัวเป็น 224px → แว่น ~90 px ✅ |
| การจับคู่ของกับคน | heuristic `IoU ∩ person` เดาเอา | **ไม่ต้องจับคู่** — crop มาจากคนคนนั้นอยู่แล้ว ✅ |
| "ไม่ใส่หมวก" | ต้องมีคลาส `no_helmet` แยก | เป็นแค่ label 0 ของ head ✅ |
| ต้นทุน | 1 pass | 1 pass + classifier batch (~5ms/8 คน บน MPS) |
| ปรับ PPE รายกล้อง | ต้อง filter หลังบ้าน | ปิด output head ได้ตรงๆ ✅ |

**ข้อแลกเปลี่ยนที่ต้องพูดตรงๆ:** ต้อง label ข้อมูลใหม่เกือบทั้งหมด เพราะข้อมูลเดิมใช้ไม่ได้
(ไม่มีกรอบคนคู่กับของ) นี่คือต้นทุนจริงของแผนนี้ และเป็นสิ่งที่หลีกเลี่ยงไม่ได้ไม่ว่าจะเลือกสถาปัตยกรรมไหน

## Pipeline การเทรน — 9 ขั้น แต่ละขั้นคุณตรวจก่อนไปต่อ

> **กฎ:** ผมจะ**ไม่เริ่มขั้นถัดไป**จนกว่าคุณจะเซ็นรับ Gate ของขั้นปัจจุบัน
> ทุก Gate ส่งมอบ 3 อย่างเสมอ: **① ตัวเลขบน ZENTRA-REAL ② contact sheet 50 ภาพ (TP/FP/FN) ③ คลิปรัน 3 นาที**

---

### ✅ Stage 0 — สร้างเครื่องวัดก่อน (ยังไม่เทรนอะไรทั้งสิ้น) — **ทำเสร็จแล้วบางส่วน**

> ถ้าข้ามขั้นนี้ ทุกตัวเลขหลังจากนี้คือการหลอกตัวเอง

**0.1 แก้ leakage** ✅ — `scripts/build_person_dataset.py` แบ่ง split ใหม่**ตามภาพต้นฉบับ**
(hash ของชื่อ base → ภาพ augment ทุกใบของต้นฉบับเดียวกันตกอยู่ split เดียวกันเสมอ ไม่ขึ้นกับลำดับการอ่านไฟล์)
พิสูจน์แล้วว่า train↔val↔test ไม่มีต้นฉบับซ้ำกันเลย

เก็บเฉพาะเฟรมที่**มีกล่อง `person` จริง** — เฟรมที่ไม่มีกล่องคนไม่ได้แปลว่า "ไม่มีคนอยู่ในภาพ"
(คลาสอื่นถูก label มาจากภาพสินค้าที่ไม่เคย label คนเลย) ถ้าใส่เข้าไปเป็น background
เท่ากับ**สอนโมเดลให้มองข้ามคน**โดยตรง

`--one-per-source` ทิ้งภาพ augment ที่ Roboflow อบมาแล้ว (~3.2 ใบ/ต้นฉบับ) เพราะ ultralytics
augment ใหม่ทุก epoch อยู่แล้ว → ข้อมูลเท่าเดิม เร็วขึ้น ~3.2 เท่า

```
person_v1 (leak-free):  train 3,760 · val 807 · test 800 ภาพ   (8,960 / 1,766 / 1,958 คน)
```

**0.2 สร้าง `ZENTRA-REAL`** ⬜ **ยังไม่ได้ทำ — ต้องใช้คุณ**
- 500 เฟรม สุ่มจากกล้องจริง ครอบคลุม: กะเช้า/กะดึก, คนเดี่ยว/คนกลุ่ม (≥5 คน), ใกล้/ไกล, ยืน/ก้ม/นั่ง
- **ห้ามใช้ชุดนี้เทรนเด็ดขาด** ล็อกไว้ แตะได้เฉพาะตอนวัดผล
- 🙋 ผมเขียน labeling tool ให้ได้

> ⚠️ **ข้อจำกัดที่ต้องพูดตรง ๆ:** ตัวเลข Gate 1 ด้านล่างวัดบน**โดเมนของ Roboflow** (ภาพจากอินเทอร์เน็ต)
> ไม่ใช่กล้องโรงงานของคุณ มันบอกได้ว่า "โมเดลดีขึ้นไหม" แต่**ยังบอกไม่ได้ว่าจะทำงานได้ดีแค่ไหนหน้างานจริง**
> จนกว่าจะมี ZENTRA-REAL

**0.3 วัด baseline ที่ซื่อสัตย์** ✅ — `yolo11s` COCO (ยังไม่ fine-tune) บน val ที่ไม่รั่ว:

| | mAP50 | mAP50-95 | Precision | Recall |
|---|---|---|---|---|
| yolo11s COCO @640 | **0.598** | 0.305 | 0.727 | **0.623** |

Recall 0.62 คือตัวเลขที่สำคัญที่สุด — **พลาดคนที่ label ไว้ไป 38%** และ PPE / Zone / Fall
ยืนอยู่บนกล่องคนทั้งหมด คนที่ตรวจไม่เจอจะไม่มีตัวตนสำหรับทั้งระบบ

**0.4 วัดความเร็วเครื่อง** ✅ — val 2,713 ภาพ = 188 วิ บน M4 MPS (infer 21ms + NMS 26ms/ภาพ)

| ✅ **Gate 0** | leakage แก้แล้ว + baseline ซื่อสัตย์ + เวลา/epoch วัดแล้ว · ⬜ ค้าง: ZENTRA-REAL |
|---|---|

### 👤 Stage 1 — Person Detection (รากฐานของทุกอย่าง) — **กำลังทำ**

PPE, Zone, Fall **ทั้งหมดยืนบนกล่องคน** ถ้าขั้นนี้ไม่แม่น ที่เหลือแม่นไม่ได้

วัด baseline ก่อนแล้ว (Stage 0.3): COCO `yolo11s` ได้ mAP50 = 0.598 → **ต่ำกว่าเกณฑ์ 0.75 จริง**
จึงมีเหตุผลให้ fine-tune ไม่ใช่เทรนเพราะ "อยากเทรน"

```bash
python3 scripts/train_person.py --epochs 40 --patience 10
```

- fine-tune `yolo11s` บน `person_v1` (3,760 ภาพ, คลาสเดียว)
- ประเมินที่ **ทั้ง imgsz 640 (เทรน) และ 960 (`cfg.PERSON_IMGSZ` ที่ deploy จริง)** — สองค่านี้ต่างกัน
  ตัวเลขที่ res หนึ่งบอกอะไรเกี่ยวกับอีก res หนึ่งไม่ได้
- **แตะ test split ครั้งเดียวตอนจบ** ไม่ใช้เลือกอะไรทั้งสิ้น

| ✅ **Gate 1 (เกณฑ์ที่ตกลงกัน)** | **mAP50 ≥ 0.75** ที่ imgsz ตอน deploy · recall ต้องดีขึ้นจาก 0.623 อย่างมีนัย · คุณตรวจภาพตัวอย่างแล้วพอใจ |
|---|---|

> Gate เดิมเขียนว่า "recall ≥ 0.90 บน ZENTRA-REAL" ซึ่ง**วัดไม่ได้ตอนนี้** เพราะ ZENTRA-REAL ยังไม่มี
> เกณฑ์ข้างบนคือสิ่งที่วัดได้จริงวันนี้ ส่วน recall ≥ 0.90 บนกล้องจริงยังเป็นเป้าหมายที่ค้างอยู่

---

### 📐 Stage 2 — Zone (ไม่มีการเทรน — เรขาคณิตล้วน)

โซนไม่ใช่โมเดล มันคือ point-in-polygon บนจุดเท้า ความเสี่ยงคือ**จุดเท้าผิด**เมื่อคนถูกบังหรือถูกตัดขอบภาพ

- label 200 เฟรมว่าคนไหนอยู่ในโซน/นอกโซน (ground truth)
- วัด precision/recall ของการเข้าโซน + **latency การยืนยัน**
- ถ้าจุดเท้าเพี้ยน: เสนอใช้ homography หรือใช้ keypoint ข้อเท้าจาก pose model ที่มีอยู่แล้ว

| ✅ **Gate 2** | zone precision ≥ 0.95 · recall ≥ 0.95 บน 200 เฟรม · false entry = 0 ในคลิป 10 นาที |
|---|---|

---

### 🪖 Stage 3 — หมวกนิรภัย (helmet) — **เทรน PPE ตัวแรก**

เริ่มที่หมวกเพราะ**ข้อมูลดีที่สุด**: 28,712 ภาพ (46% มีกรอบคน) + no_helmet 3,558 ภาพ (97% มีกรอบคน)
เป็นคลาสเดียวที่ข้อมูลเดิมใช้ต่อได้จริง

- สร้างชุด crop: จากภาพที่มีทั้ง `person` และ `helmet`/`no_helmet` → crop คน → label หัว: `helmet` / `no_helmet` / `unknown`
- เทรน classifier เล็ก (เช่น `yolo11n-cls` หรือ EfficientNet-B0) input 224
- **เทรนแค่ head เดียวก่อน** ยังไม่ยุ่งกับคลาสอื่น
- ปรับ threshold จาก precision-recall curve บน ZENTRA-REAL ไม่ใช่เดา 0.30

| ✅ **Gate 3** | helmet recall ≥ 0.92 · no_helmet recall ≥ 0.95 (พลาดคนไม่ใส่หมวก = อันตราย) · FP ≤ 0.05 · ทดสอบสด 3 นาที ไม่มีป้ายกระพริบ |
|---|---|

---

### 🦺 Stage 4 — เสื้อกั๊กสะท้อนแสง (vest)

ข้อมูลเดิมมี 15,888 ภาพ แต่ **0% มีกรอบคน** → ใช้ต่อไม่ได้ตรงๆ
- เก็บข้อมูลใหม่จากโรงงาน หรือใช้ auto-label: รัน person detector บนภาพ vest เดิม ถ้าเจอคนก็กู้มาใช้ได้บางส่วน (ต้องตรวจด้วยตา)
- เสื้อกั๊กมีสีเด่นชัด → คาดว่าง่ายที่สุดรองจากหมวก
- เพิ่ม head `vest` เข้า classifier เดิม (multi-label) แล้ว **วัดว่า helmet ตกลงหรือไม่** (regression check)

| ✅ **Gate 4** | vest recall ≥ 0.90 · no_vest recall ≥ 0.92 · **helmet ไม่ตกจาก Gate 3 เกิน 2%** |
|---|---|

---

### 🧤 Stage 5 — ถุงมือ (gloves)

ข้อมูลเดิม 2,260 ภาพ **โคลสอัพมือล้วน 0% มีคน** → **ทิ้ง เริ่มใหม่**
- crop บริเวณมือจาก keypoint ข้อมือ (`Left/Right Wrist`) ของ `yolo11n-pose` ที่มีอยู่แล้ว
- ได้ crop มือ ~64–96 px → classify `gloves` / `no_gloves` / `hand not visible`
- ต้องการ ~2,000 crop มือจากกล้องจริง

| ✅ **Gate 5** | gloves recall ≥ 0.80 · abstain rate รายงานชัด (มือถูกบังบ่อย) · ไม่ทำให้ Gate 3/4 ตก |
|---|---|

---

### 👓 Stage 6 — แว่นตานิรภัย (glasses) — **ยากที่สุด**

ข้อมูลเดิม: glasses 1,264 ภาพ / no_glasses **668 ภาพ** ทั้งหมดโคลสอัพ → ทิ้ง
- crop หัวจาก keypoint จมูก/ตา → 96–128 px
- **ถ้าหัวคนในกล้องเล็กกว่า ~40 px แว่นตรวจไม่ได้ในทางฟิสิกส์** — ต้องยอมรับและ **abstain** แทนที่จะเดา
- กำหนดระยะทำงานสูงสุดจากการวัด แล้วบอกผู้ใช้ตรงๆ ในหน้า Settings

| ✅ **Gate 6** | glasses recall ≥ 0.70 **ในระยะที่ประกาศไว้** · FP ≤ 0.05 · นอกระยะต้อง abstain ไม่ใช่เดา |
|---|---|

---

### 🥾 Stage 7 — รองเท้านิรภัย (boots)

ข้อมูลเดิม 704 ภาพ และ **`no_boots` มี 0 ตัวอย่างใน valid → วัดผลไม่ได้เลย**
- crop เท้าจาก keypoint ข้อเท้า
- เท้ามักถูกบังด้วยโต๊ะ/ของ → abstain rate จะสูง เป็นเรื่องปกติ
- ถ้าเก็บข้อมูลไม่พอ **ผมจะเสนอให้ตัดคลาสนี้ออก** ดีกว่าปล่อยโมเดลที่เดามั่ว

| ✅ **Gate 7** | boots recall ≥ 0.75 หรือ **ตัดสินใจร่วมกันว่าจะตัดทิ้ง** |
|---|---|

---

### 🚑 Stage 8 — Fall Detection

ตอนนี้ใช้ TFLite Transformer seq-30 (MIT) ที่ **เทรนมาจากคนเดียว** และเราไม่ได้เทรนเอง

- **8.1** เก็บข้อมูล: ล้ม 20 ครั้ง (หน้า/หลัง/ข้าง, ใกล้/ไกล, มีสิ่งบัง, 2 คนพร้อมกัน) + ไม่ล้ม 10 นาที (เดิน ก้ม **นั่งกับพื้น** คลาน)
- **8.2** วัด baseline ของโมเดลปัจจุบัน: recall / precision / latency
- **8.3** ถ้า recall < 0.90 → เทรน seq-60 ใหม่ ด้วย **feature format เดิมทุกประการ** (ไม่ต้องแก้โค้ดฝั่ง inference เลย)
- ⚠️ **ช่องโหว่ที่รู้อยู่**: ไม่มีคลิป "นั่งกับพื้นโดยไม่ได้ล้ม" ในชุดทดสอบเลย → FP rate ของท่านี้ยังไม่เคยวัด

| ✅ **Gate 8** | fall recall ≥ 0.90 · precision ≥ 0.95 · latency ≤ 4 วิ · **นั่งกับพื้น 5 นาที = 0 FP** |

## สิ่งที่ผมยังไม่รู้ และจะไม่เดา

1. **เวลาเทรนต่อ epoch บน M4** — วัดใน Stage 0.4 ไม่ประมาณ
2. **ประสิทธิภาพจริงในโรงงานคุณ** — ไม่มีใครรู้จนกว่าจะมี ZENTRA-REAL
3. **crop classifier จะดีกว่าจริงไหม** — เป็นเหตุผลเชิงฟิสิกส์ (ขนาด px) ที่หนักแน่น แต่ยัง**ไม่ได้วัด** จะพิสูจน์เทียบกันใน Stage 3 ก่อนผูกมัดทั้งระบบ
4. **จำนวนภาพที่ต้อง label** — ประมาณ 500 (test) + ~2,000/คลาสใหม่ ปรับได้หลังเห็น learning curve
